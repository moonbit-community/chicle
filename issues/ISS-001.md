# ISS-001: Populate layout order for positioned children

## Description
`Layout.order` is part of the public layout result and is intended to describe the relative rendering order of nodes. The pinned Rust Taffy implementation assigns source order to block, flex, and grid children, while the MoonBit implementation only initializes `order` to `0` in `Layout::zero()` and never overwrites it.

A runtime probe with a root and two leaf children currently reports `orders root=0 a=0 b=0`. This makes the public field unusable for consumers that rely on Taffy layout output to derive stable paint order.

## Design
Propagate source order at the point where each layout algorithm performs final child placement, matching the pinned Rust Taffy v0.5.2 behavior. Keep the root order at zero. Cover block, flex, grid, absolute-positioned children, hidden nodes, and cache restoration. Do not introduce a new z-index feature; this issue concerns the existing Taffy source-order contract only.

## Acceptance Criteria
- [x] Sibling layouts receive the same relative `order` values as pinned Rust Taffy v0.5.2 for block, flex, grid, and absolute-positioned placement.
- [x] The root layout retains `order == 0`.
- [x] Recomputed and cache-restored layouts preserve the assigned order.
- [x] Focused downstream-style regression tests fail before the fix and pass after it.
- [x] `moon check`, `moon check --warn-list +unnecessary_annotation`, and `moon test --no-render --target all` pass.
