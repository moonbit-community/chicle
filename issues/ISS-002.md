# ISS-002: Include overflowing descendants in container content size

## Metadata

## Description
For non-leaf nodes, `finalize_layout_metadata` currently derives `Layout.content_size` from the container border-box size minus padding, border, and scrollbar insets. It does not accumulate the spatial contribution of overflowing children. Rust Taffy defines `content_size` as the content extent inside a node, which may exceed the node size and is used to compute scroll width and height.

A runtime probe with a 100x50 block container and a 200x20 block child reports `root size=100x50 content=100x50 scroll=0x0` while the child layout remains 200px wide. The expected horizontal content extent is 200px, yielding a non-zero scroll width according to the pinned upstream semantics.

## Design
Port the upstream content-size contribution model for in-flow and absolute children across block, flex, and grid algorithms. Respect overflow clipping semantics, child location, child size, descendant content size, scrollbars, padding, and borders. Keep leaf measurement behavior intact. Avoid computing content extent only as a post-hoc maximum of raw child sizes because positioned offsets and nested overflow affect the result.

## Acceptance Criteria
- [ ] A 200px-wide visible-overflow child inside a 100px-wide block container produces the upstream-equivalent `content_size` and positive `scroll_width()`.
- [ ] Overflow contributions are correct for block, flex, grid, absolute-positioned children, nested overflow, and negative or positive child offsets.
- [ ] Hidden and clipped overflow do not incorrectly enlarge the reported scrollable content extent.
- [ ] Existing leaf measurement, scrollbar, border, and padding metadata behavior remains unchanged.
- [ ] Focused downstream-style regression tests fail before the fix and pass after it.
- [ ] `moon check`, `moon check --warn-list +unnecessary_annotation`, and `moon test --no-render --target all` pass.
